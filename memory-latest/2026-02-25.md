## Claude Code CLI 实测结论
- 简单任务(echo/读文件/创建小文件)：✅ ~15s，能用
- 复杂任务(500行JSON/多文件修改)：❌ 超时无输出，即使给180s也不行
- 原因：中转站延迟 + 大量 token 输出 + pty 模式捕获不稳定
- **结论：Claude Code 只适合小任务，复杂编码还是用 subagent**

## Discord 图片识别故障排查 (进行中)
- **根因**: OpenClaw 2026.2.23 版本新增了 RFC2544 (198.18.0.0/15) SSRF 拦截，而 OpenClash fake IP 模式正好用这个段
- Changelog 明确写了: "Security/SSRF: block RFC2544 benchmarking range (198.18.0.0/15)"
- Telegram 有 `allowRfc2544BenchmarkRange` opt-in，但 Discord 没有，config schema 也不暴露这个字段
- **修复方案**: 直接 patch OpenClaw 源码 `ssrf-*.js` 文件，把 `resolveIpv4SpecialUseBlockOptions` 返回值改为 `{ allowRfc2544BenchmarkRange: true }`
- patch 文件路径: `/Users/yunhan/.nvm/versions/node/v24.13.0/lib/node_modules/openclaw/dist/ssrf-*.js`（有 .bak 备份）
- **注意**: OpenClaw 升级会覆盖 patch，需要重新打
- **当前状态**: patch 已打，但旧 gateway 进程(pid 78220)没被杀掉，新进程启动失败。需要 kill -9 旧进程再重启

## Gateway 重启经验教训
- `gateway stop` 会卸载 LaunchAgent，之后必须 `gateway install` 再 `start`
- 正确流程: `openclaw gateway install && openclaw gateway start`
- 如果端口被占（旧进程残留），需要 `kill -9 <pid>` 后再启动
- 新版日志路径: `~/.openclaw/logs/gateway.log` 和 `gateway.err.log`
